# 4 РАЗРАБОТКА САЙТА

В данной главе рассматривается практическая реализация веб-платформы peer-to-peer аренды «Рентли»: от выбора средств разработки и организации данных до описания пользовательского интерфейса и ключевых сценариев — каталог объявлений, личный кабинет, согласование аренды и обмен сообщениями между пользователями. В учебной реализации в шапке интерфейса используется рабочее название RentSpace; в пояснительной записке платформа именуется «Рентли».

Настоящий файл содержит **только материалы к главе 4** пояснительной записки. Остальные главы (введение, аналитический раздел, проектирование, экономика, заключение, приложения) оформляются отдельными документами или разделами общего файла Word и **не удалялись** — они просто не входят в этот фрагмент, чтобы не дублировать текст всего диплома в одном месте.

## 4.1 Описание инструментов разработки

Для создания веб-платформы аренды были применены современные технологии, обеспечивающие разделение клиентской и серверной частей и удобную доработку функциональности по мере развития проекта. Клиентская часть реализована как одностраничное приложение: после первой загрузки страница не перезагружается целиком при переходе между каталогом, карточкой товара и личным кабинетом, а данные для списков и форм запрашиваются у сервера по протоколу HTTP в формате JSON. Такой подход соответствует ожиданиям пользователей, привыкших к динамическим интерфейсам маркетплейсов, и снижает нагрузку на сеть за счёт отсутствия полной перерисовки документа при каждом действии.

Основой клиента выбрана библиотека React, которая позволяет описывать интерфейс в виде компонентов и при необходимости наращивать экраны — профиль, избранное, управление заявками на аренду — без полной перестройки приложения. Компонентный подход упрощает тестирование отдельных частей интерфейса и повторное использование элементов: например, модальные окна авторизации, формы и списки подключаются в разных сценариях без копирования разметки. Сборка и локальная отладка выполняются средствами Vite: быстрый запуск среды разработки и оптимизированная производственная сборка сокращают время итераций при правках вёрстки и логики. Для контроля качества клиентского кода используется линтер ESLint в составе проекта; это помогает выявлять типичные ошибки до запуска приложения в браузере.

Навигация между разделами реализована средствами React Router без полной перезагрузки документа в браузере. Обмен данными с сервером выполняется через библиотеку Axios; для работы с сессионной аутентификацией настроена передача учётных данных запроса вместе с cookie, что согласуется с механизмом входа пользователя на стороне Django. Таким образом, после успешного входа последующие запросы к защищённым ресурсам выполняются в контексте уже установленной сессии без ручной подстановки токена в заголовки на стороне клиента.

Стилизация выполнена комбинированно: глобальные правила вынесены в таблицы стилей приложения, отдельные экраны и модальные окна дополняются объектными стилями в коде компонентов, что даёт точный контроль над отступами и сеткой без конфликта имён классов. В ряде компонентов дополнительно формируются встроенные стили для медиа-запросов, чтобы адаптировать шапку и сетки к мобильным ширинам экрана. К проекту подключены инструменты постобработки стилей в составе цепочки сборки; при необходимости утилитарные классы могут дополнять вёрстку без переписывания всей разметки.

Серверная часть построена на языке Python с использованием фреймворка Django и надстройки Django REST Framework. Объектно-реляционное отображение Django позволяет описать предметную область — пользователя и профиль, категории и товары, аренду, сообщения, уведомления, отзывы и избранное — в виде моделей и сопровождать изменения схемы миграциями. Зависимости серверной части фиксируются в файле требований виртуального окружения; это обеспечивает воспроизводимость стенда на другой машине. Слой REST API обеспечивает единообразный доступ к данным через сериализаторы и настройки прав доступа. Для совместной работы фронтенда и бэкенда на разных портах локальной машины использованы механизм разрешения межсайтовых запросов и настройки защиты от подделки межсайтовых запросов для доверенных источников, чтобы браузер не блокировал обращения из клиентского приложения к серверу при учебной схеме развёртывания.

Разработка велась в редакторе Visual Studio Code; проверка запросов и ответов выполнялась инструментами разработчика в браузере (вкладки «Сеть» и «Консоль») и журналом сервера Django. Для анализа структуры компонентов React при отладке удобно использовать соответствующее расширение инструментов разработчика, позволяющее просматривать дерево компонентов и состояние контекста.

Состав зависимостей клиента и сценарии сборки задаются в манифесте пакета: команда запуска среды разработки, сборка production-версии и проверка линтером. На стороне сервера в настройках проекта подключаются приложения Django REST Framework и посредник для межсайтовых запросов, задаётся движок СУБД и параметры REST по умолчанию (в том числе сессионная аутентификация для API). ## 4.2 Проектирование базы данных

Для хранения данных учебного стенда выбрана встраиваемая СУБД SQLite: один файл базы упрощает перенос проекта между компьютерами и не требует отдельного сервера баз данных, при этом возможностей достаточно для отладки сценариев аренды, коммуникации пользователей и работы каталога. Для учебного проекта это разумный компромисс между простотой установки и полнотой функций реляционной модели. При переходе к промышленной эксплуатации предполагается использование PostgreSQL без смены моделей Django — достаточно изменить параметры подключения в настройках проекта и при необходимости скорректировать особенности развёртывания (пул соединений, резервное копирование), не переписывая бизнес-логику.

Логическая структура отражает связи предметной области. Учётная запись пользователя в стандартной модели Django дополняется профилем «один-к-одному» с полями телефона, аватара, области и числового рейтинга; профиль создаётся автоматически при регистрации пользователя сигналами фреймворка. Категории товаров поддерживают иерархию: родительская категория и при необходимости подкатегория, что позволяет строить многоуровневый справочник без дублирования названий. Товар привязан к владельцу и к категории, может иметь подкатегорию, содержит текстовое описание, цену за день, залог и статус доступности (например, доступен, арендован, на обслуживании). Для карточки товара поддерживается как базовое изображение, так и отдельная коллекция медиафайлов (`ItemMedia`) для отображения галереи. Запись об аренде связывает товар, арендатора, период дат, рассчитанную сумму и статус жизненного цикла заявки — от ожидания подтверждения до завершения или отмены.

Отдельно хранятся сообщения между пользователями с возможностью текстового содержимого и вложений (изображение или файл), признаком прочитанности и временем создания. Уведомления содержат тип события, заголовок, текст, связь с товаром и арендой при необходимости, чтобы из интерфейса можно было перейти к контексту. Отзывы связывают автора, оцениваемого пользователя, товар и аренду; предусмотрено ограничение уникальности по паре автор–аренда, чтобы не допускать множественных отзывов на одну сделку. Избранное хранит пары пользователь–товар с уникальностью по этой паре. В совокупности перечисленные сущности поддерживают коммуникацию, доверие и повторные обращения к платформе.

Создание и изменение таблиц выполняется стандартными командами миграций Django, что фиксирует историю эволюции схемы в репозитории и позволяет воспроизвести ту же структуру на другой машине. ## 4.3 Серверное приложение и программный интерфейс

Центральная точка маршрутизации веб-сервиса задаётся в конфигурации URL проекта Django. К префиксу маршрутов API подключается роутер Django REST Framework, который регистрирует набор контроллеров-наборов представлений для категорий, товаров, аренд, сообщений, уведомлений, отзывов и избранного. Такой подход соответствует стилю REST: ресурсы именуются существительными, а специальные операции (например, список диалогов или переписка с одним пользователем) выносятся в дополнительные действия контроллера. Отдельно подключены представления аутентификации: регистрация, вход, выход, профиль текущего пользователя, списки собственных аренд и товаров, а также агрегированная статистика по пользователю. Разделение «ресурсы каталога» и «сессия пользователя» упрощает понимание структуры API при сопровождении проекта.

Сериализаторы задают формат входных и выходных данных и служат посредником между моделями ORM и JSON-ответом. Для товаров вычисляются производные поля с именем владельца, рейтингом, аватаром и областью владельца; дополнительно возвращается массив `media` для галереи изображений. На запись поддерживается пакетная загрузка файлов через поле `media_files`, после чего сервер создаёт связанные записи медиа. Для сообщений при наличии вложений формируются полные адреса файлов с учётом текущего запроса. Правила доступа настраиваются так, чтобы просмотр каталога и отдельных объявлений был доступен без входа, а создание аренды, сообщений и отзывов — только авторизованным пользователям. Логика смены статусов аренды реализована отдельными действиями контроллера аренды: при переходах между состояниями обновляются поля товара (например, «арендован» при подтверждении и снова «доступен» при завершении) и создаются записи уведомлений для второй стороны сделки. Для экрана управления арендой используется параметр выборки `as_owner=true`, чтобы в интерфейс попадали только заявки по товарам текущего владельца. Раздача загруженных изображений товаров и аватаров в режиме разработки выполняется через настройки медиакаталога; для промышленного развёртывания рекомендуется отдавать статику и медиа через веб-сервер или внешнее хранилище, а не встроенный сервер разработки.

Для товаров реализованы дополнительные действия: выдача периодов занятости по активным арендам для отображения календаря на клиенте и формирование рекомендаций (похожие по категории и товары, которые выбирали те же арендаторы). Для контроллера отзывов при создании записи пересчитывается средний рейтинг профиля оцениваемого пользователя и создаётся уведомление. Для избранного контроллер возвращает список товаров в формате, совместимом с карточками каталога.

Фрагмент регистрации маршрутов API приведён ниже.

Пример обработки входа: представление принимает данные формы входа, проверяет их через сериализатор, при успехе создаёт сессию Django и возвращает клиенту сериализованного пользователя с полями профиля.

```python
class LoginView(APIView):
    permission_classes = [AllowAny]

    def post(self, request):
        serializer = UserLoginSerializer(data=request.data)
        if serializer.is_valid():
            user = serializer.validated_data['user']
            login(request, user)
            user_serializer = UserSerializer(user)
            return Response({
                'success': True,
                'user': user_serializer.data
            }, status=status.HTTP_200_OK)
        return Response({
            'success': False,
            'errors': serializer.errors
        }, status=status.HTTP_400_BAD_REQUEST)
```

## 4.4 Организация файлов проекта и клиентская часть

Корень решения разделён на каталог серверной части с проектом Django и приложением каталога и на каталог клиентского приложения, собираемого Vite. На стороне сервера в типичной структуре Django выделяются настройки проекта, корневой файл URL, приложение с моделями и представлениями, а также статические файлы при сборке. На стороне клиента исходный код сосредоточен в каталоге с компонентами экранов и модальных окон, отдельным каталогом контекста аутентификации и точкой входа, которая монтирует корневой компонент в документ страницы; в корневом компоненте приложения подключается провайдер контекста аутентификации и задаётся маршрутизация к главной странице с каталогом, странице товара и личному кабинету. Общие элементы интерфейса (шапка и подвал) подключаются в общий макет, чтобы не дублировать разметку на каждом маршруте.

В актуальной версии клиентской части подвал реализован в корневом компоненте `App.jsx` и отображается на всех основных страницах. В его составе предусмотрены навигационные ссылки («Главная лента», «Личный лор»), внешняя ссылка на `max.ru` и строка копирайта `© 2026 Рентли`. Такое решение стандартизирует нижнюю область интерфейса и обеспечивает единообразную навигацию независимо от текущего маршрута.

Среди ключевых компонентов клиентской части можно назвать список объявлений и карточку товара, форму аренды и боковую панель фильтров, модальные окна авторизации, уведомлений и переписки, компоненты личного кабинета с вкладками и управлением заявками. Такая модульность облегчает поиск ошибок: проблема в отображении каталога с высокой вероятностью локализуется в компоненте списка или в запросе данных, а не во всём приложении целиком. ## 4.5 Главная страница: шапка, поиск и каталог

### Общее оформление и композиция экрана

Визуальное решение главной страницы выдержано в единой цветовой гамме с градиентным фоном и контрастными светлыми карточками объявлений. Такое оформление задаёт узнаваемый характер интерфейса маркетплейса и одновременно сохраняет читаемость текста и кнопок. Основная область под шапкой разделена на две колонки: слева располагается узкая боковая панель с фильтрами, справа — вертикальная колонка с промо-баннером и сеткой карточек. На широком экране такая композиция позволяет одновременно видеть условия отбора и результат; при уменьшении ширины окна элементы могут перестраиваться за счёт встроенных стилей и гибких сеток.

### Шапка, поиск и навигация

При запуске приложения пользователь видит шапку сайта, закреплённую у верхней границы окна: логотип, поле поиска по объявлениям, ссылки на главную, выпадающее меню категорий и справочный раздел о том, как работает аренда, а также блок входа или, после авторизации, кнопки добавления товара, перехода к уведомлениям и сообщениям и ссылку в личный кабинет. Шапка выполнена в стиле «стеклянной» панели с размытием фона и адаптируется к узким экранам за счёт медиа-запросов, встроенных в компонент: на мобильных устройствах часть навигации может скрываться или переноситься, чтобы сохранить удобство нажатия по кнопкам.

Поиск по каталогу реализован через подстановку параметров в адрес главной страницы и последующую фильтрацию списка на стороне клиента по названию, описанию и при необходимости названию категории. Такой режим не требует отдельного поискового сервиса на этапе учебного прототипа и даёт предсказуемое поведение при демонстрации. Меню категорий в шапке запрашивает у сервера список родительских категорий; для наглядности рядом используются подсказки по подкатегориям и иконкам, что облегчает ориентацию пользователя в тематике аренды (техника, инструменты, мероприятия и т.д.). Выбор категории отражается в адресе и синхронизируется с боковой панелью фильтров.

Счётчик непрочитанных уведомлений для авторизованного пользователя запрашивается при появлении шапки и затем обновляется с заданным интервалом; рядом с пунктом «Уведомления» отображается числовой бейдж, если есть непрочитанные события. На сервере подсчитывается количество записей уведомлений, помеченных как непрочитанные для текущего пользователя. Аналогично открывается модальное окно переписки: пользователь может перейти к диалогам без смены страницы.

Отправка формы поиска в шапке переводит пользователя на главную страницу с подстановкой строки запроса в адрес; выбор категории из меню также задаёт параметры маршрута. Ниже показан фрагмент обработчиков.

Каталог дополнительно поддерживает фильтр по области и группировку объявлений по областям владельцев, что важно для сценария аренды в пределах конкретного региона. ### Структура карточки объявления

Страница объявления оформлена в виде крупного блока с медиа-галереей и соседней колонкой с текстом на широком экране; при необходимости сетка перестраивается в одну колонку на узких дисплеях. Такое расположение позволяет быстро оценить условия сделки и уровень доверия к владельцу до открытия переписки.

### Оформление аренды, избранное и связь с владельцем

Предусмотрена кнопка оформления аренды: для гостя выводится напоминание о необходимости входа, для авторизованного пользователя открывается модальное окно с выбором периода и расчётом суммы. Рядом расположена кнопка избранного: состояние проверяется при загрузке страницы и переключается запросами к API избранного. ### Календарь занятости и рекомендации

Для наглядности выводится календарь занятости: на сервере выбираются аренды по товару в статусах, соответствующих незавершённой сделке, и передаются клиенту для отображения занятых дат и текущего дня. Это помогает арендатору визуально сопоставить желаемый период с уже занятыми интервалами.

Ниже основного содержимого выводятся рекомендации. Алгоритм на стороне сервера формирует два набора: во-первых, похожие товары из той же категории со статусом доступности; во-вторых, товары, которые выбирали пользователи, ранее арендовавшие рассматриваемое объявление — эвристика «с этим также арендовали» повышает полезность каталога без сложной системы коллаборативной фильтрации. Карточки рекомендаций кликабельны и ведут на соответствующие страницы товаров.

```python
@action(detail=True, methods=['get'])
def recommendations(self, request, pk=None):
    item = self.get_object()
    similar_items = Item.objects.filter(
        category=item.category,
        status='available'
    ).exclude(id=item.id).order_by('-created_at')[:8]
    renter_ids = Rental.objects.filter(item=item).values_list('renter_id', flat=True)
    also_rented_items = Item.objects.none()
    if renter_ids:
        also_rented_items = (
            Item.objects.filter(
                rental__renter_id__in=renter_ids,
                status='available'
            )
            .exclude(id=item.id)
            .order_by('-created_at')[:8]
        )
    return Response({
        'similar': ItemSerializer(similar_items, many=True, context=self.get_serializer_context()).data,
        'also_rented': ItemSerializer(also_rented_items, many=True, context=self.get_serializer_context()).data,
    })
```

## 4.7 Личный кабинет

Личный кабинет открывается отдельным маршрутом и оформлен как полноэкранная страница с вкладками и крупной шапкой профиля на градиентном фоне. В верхней части отображаются аватар, имя пользователя и сводные числа: количество аренд, количество опубликованных товаров и рейтинг. В текущей версии применяется единый тип пользователя без разделения на роли, поэтому один и тот же аккаунт может одновременно публиковать товары и арендовать чужие позиции.

Вкладка профиля показывает логин, адрес электронной почты, телефон, область и рейтинг; предусмотрена кнопка редактирования, открывающая модальное окно с формой обновления данных (включая аватар и область). На этапах регистрации и редактирования профиля область выбирается из полного списка субъектов Российской Федерации. Раздел «Мои аренды» получает с сервера список сделок пользователя как арендатора; карточки содержат название товара, период дат, сумму, дату создания заявки и статус с цветовой меткой (ожидание, подтверждение, активная сделка, завершение, отмена). При отсутствии записей выводится пустое состояние с предложением перейти в каталог.

Вкладки «Мои товары», «Управление арендами», «Статистика» и «Избранное» доступны всем авторизованным пользователям. ## 4.8 Обмен сообщениями и центр уведомлений

### Сообщения

Модуль сообщений реализован как модальное окно поверх основного интерфейса, чтобы пользователь не терял контекст текущей страницы. В левой части отображается список диалогов: для каждого собеседника показываются имя, фрагмент последнего сообщения, время и счётчик непрочитанного при наличии. В правой части — лента переписки с выбранным пользователем в хронологическом порядке. Список диалогов формируется на сервере на основе множества сообщений текущего пользователя; при первом открытии окна без явного выбора может автоматически подставляться последний активный диалог для ускорения работы.

При открытии переписки входящие сообщения от выбранного собеседника помечаются прочитанными, чтобы счётчики и индикаторы не задерживали устаревшее состояние. Отправка выполняется запросом с телом в формате «многократной формы», что позволяет передавать не только текст, но и вложения — изображение или произвольный файл, что важно для уточнения состояния техники или реквизита по фотографии. Для обновления ленты без постоянного сетевого соединения в учебном проекте используется периодический опрос с заданным интервалом; в промышленной версии этот слой можно заменить или дополнить технологией постоянного соединения. При создании нового сообщения получателю создаётся уведомление, согласованное по типу с остальными событиями платформы.

Список диалогов на сервере строится по множеству сообщений текущего пользователя: определяются уникальные собеседники, для каждого подбирается последнее сообщение и число непрочитанных входящих.

```python
@action(detail=False, methods=['get'])
def conversations(self, request):
    user = request.user
    sent_to = Message.objects.filter(sender=user).values_list('recipient', flat=True).distinct()
    received_from = Message.objects.filter(recipient=user).values_list('sender', flat=True).distinct()
    all_users = set(list(sent_to) + list(received_from))
    conversations = []
    for user_id in all_users:
        other_user = User.objects.get(id=user_id)
        last_message = Message.objects.filter(
            models.Q(sender=user, recipient=other_user) |
            models.Q(sender=other_user, recipient=user)
        ).order_by('-created_at').first()
        unread_count = Message.objects.filter(
            sender=other_user, recipient=user, is_read=False
        ).count()
        conversations.append({
            'user_id': other_user.id,
            'username': other_user.username,
            'last_message': MessageSerializer(last_message, context={'request': request}).data if last_message else None,
            'unread_count': unread_count
        })
    return Response(conversations)
```

## 4.9 Публикация объявления, сборка и ограничения версии

Добавление товара выполняется из шапки через модальную форму. Пользователь вводит название и описание, выбирает категорию и при необходимости подкатегорию из справочника, задаёт цену за день и залог и может прикрепить сразу несколько медиафайлов; на сервере владелец подставляется из текущей сессии, что исключает подмену автора объявления со стороны клиента. После успешного сохранения каталог на главной странице обновляется за счёт описанного ранее механизма обновления списка без перезагрузки всего приложения.

Модальное окно входа и регистрации объединяет оба сценария с переключением режима; после успешного ответа сервера данные пользователя попадают в контекст приложения и интерфейс переключается в режим авторизованного пользователя — отображаются кнопки добавления товара, уведомлений, сообщений и ссылка в личный кабинет. Выход выполняется запросом к серверу с очисткой сессии и сбросом состояния на клиенте.

Production-сборка клиента выполняется стандартной командой сборки Node в каталоге фронтенда; полученные статические файлы могут размещаться на веб-сервере или CDN. Сервер Django в учебных целях запускается встроенным сервером разработки; для эксплуатации рекомендуется связка процесса приложения с обратным прокси и, при необходимости, отдельным сервером для раздачи статики.

Учебная версия намеренно не включает ряд возможностей промышленного уровня: интеграцию с платёжными системами и проведение оплаты через платформу, юридически значимый учёт залога и эскроу, полноценный чат в реальном времени на постоянном соединении, расширенную модерацию контента и строгую верификацию личности. Перечисленные функции требуют отдельной правовой и инфраструктурной проработки. Заложенная архитектура моделей и программного интерфейса позволяет наращивать эти направления постепенно, не переписывая ядро системы целиком.

Форма публикации товара на клиенте собирает поля и при наличии файлов отправляет их в виде «многократной формы» (`multipart/form-data`), где каждый файл передаётся как `media_files`; категории подгружаются при открытии модального окна. На сервере в контроллере товаров при создании записи владелец назначается из текущей сессии, а сериализатор создаёт связанные записи медиа.
